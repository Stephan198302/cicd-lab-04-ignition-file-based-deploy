Two parts. In the first part you become the person who can point at any file in an Ignition gateway's data/ directory and say what it is, who owns it, and whether it belongs in git. PART 1 · WE DO
Add a database connection, find where it lands
The marker trick: set a timestamp marker, make the change in the UI, then ask find what's newer. Watch the connection land under config/resources/<scope>/.
1
Set the marker (docker exec lab04-ignition-local touch /tmp/marker). Re-touch it before every change you want to trace.
2
UI: Config → Databases → Connections → New. Postgres datasource → host timescaledb, port 5432, db ignition_loc, user/pass from .env (default ignition/ignition). Save.
3
Run the find -newer. The connection lands at config/resources/<scope>/ignition/database-connection/<name>/config.json. Because local bind-mounts ./services/config/, it also just appeared in your repo on the host: run git status.
4
Answer out loud: gateway-level or project-level? Would you commit it? Which scope did it land in, and why does that matter for dev vs prod?
# hostname "timescaledb" is the compose service name;
# the gateway resolves it on the lab's docker network

docker exec lab04-ignition-local touch /tmp/marker
# …make the UI change, then:
docker exec lab04-ignition-local find \
  /usr/local/bin/ignition/data/config \
  -newer /tmp/marker -type f 2>/dev/null

docker exec lab04-ignition-local cat <path-you-found>
Heads-up

Expect a longer path than 8.1 habits suggest: scope first, then module namespace. If you can't find the file, you probably forgot to re-touch the marker before the change.

Part 1 · solo · which bucket, where on disk?
Make three changes. Classify every one.

Pick three from the list (or invent your own, where at least one of your own is worth more). For each, note: what you changed · where it lives on disk (relative to data/) · which bucket · would you commit it, one sentence why.

  • A Perspective view in a project. No Designer? Simulate on disk: projects/sample/com.inductiveautomation.perspective/views/Hello/{view.json, resource.json} (minimal content in docs/ignition-file-structure.md, and the resource.json manifest is required).
  • Gateway timezone (Config → System → Time) · a new user (Config → Security → Users) · a new tag provider (Config → Tags → Realtime) · toggle a module (Config → Modules).
  • Use the marker trick every time: re-touch before each change, find -newer after. One of the five lands somewhere that must never go in git. Which?
Done when

You can find any UI change's file with the marker trick · say for each of your three changes which bucket it's in and whether you'd commit it · state the 8.3 path shape for a view and a config resource · explain why db/ and .resources/ are never committed.

File-based deploy · pre-flight
Part 2 · before anything can deploy
Pre-flight: five things your fork needs
  • Fork with Actions enabled. Forks ship with workflows disabled, so open your fork's Actions tab and click "enable" or nothing will ever trigger.
  • A develop branch, Git Flow's integration branch. Optionally make it the fork's default branch so PRs target it automatically.
  • The GitHub CLI (gh) authenticated, with RUNNER_REPO_URL pointing at your fork. setup.sh mints the runner's registration token with gh — no PAT.
  • An API key per gateway: each gateway's UI → Config → Security → API Keys → New, scoped to Project Scan + Config Scan only. Copy immediately, because you can't read it back. Into .env as IGNITION_API_KEY_LOCAL/_DEV/_PROD.
  • Two GitHub environments: lab-gateway-dev and lab-gateway-prod, set up in Part 2.
# the develop branch, one-time
git checkout -b develop && git push -u origin develop

PART 2 · WE DO
The deploy, by hand, once
Everything deploy.yml automates, done manually so the workflow holds no mysteries.
1
Edit projects/sample/…/views/Hello/view.json (e.g. width: 800 → 1200). It's already on the local gateway's disk via the bind mount.
2
Scan local, verify at :8088. The scan makes the gateway notice what's on disk.
3
Ship to dev by hand: wipe, docker cp, scan. Verify at :8089: same view, same width.
4
Open .deployignore. For each pattern, say why the gateway shouldn't have that file.
# 2 (local): files already there, just scan
scripts/scan.sh both --gateway local

# 3 (dev): wipe, copy, scan (deploy.yml does exactly this)
docker exec lab04-ignition-dev sh -c \
  "rm -rf /usr/local/bin/ignition/data/projects/* \
          /usr/local/bin/ignition/data/config/*"
docker cp ./projects/. \
  lab04-ignition-dev:/usr/local/bin/ignition/data/projects/
docker cp ./services/config/. \
  lab04-ignition-dev:/usr/local/bin/ignition/data/config/
scripts/scan.sh both --gateway dev
No sample project?

If you skipped the demo, use any view under the shipped projects/example-project/ instead. The flow is identical.

File-based deploy · you do · parts 1-2
PARTS 1-2
Runner online + environments & secrets
15 min. The plumbing between your fork and your gateways.
1
Verify the runner (it's part of the compose stack). Then check your fork: Settings → Actions → Runners must show it online with labels self-hosted, lab04.
2
Not there? Three checks: RUNNER_REPO_URL points at your fork · gh was authenticated when you ran setup (it mints the token) · docker compose restart github-runner.
3
Create environment lab-gateway-dev (Settings → Environments) with secret IGNITION_API_KEY = the key you generated on the dev gateway (:8089).
4
Repeat: lab-gateway-prod with the key from the prod gateway (:8090). Leave IGNITION_URL / IGNITION_CONTAINER unset; the defaults match the bundled runner.
# part 1: is the runner alive?
docker compose ps github-runner
docker compose logs --tail 50 github-runner
# look for "Listening for Jobs"
Per-gateway keys

API keys are per-gateway, not per-URL. The dev key won't authenticate against prod. And the runner reaches gateways by compose service name (http://ignition-dev:8088). The host ports 8089/8090 are only how you reach the UIs.

PART 3
Ship to dev via develop
15 min. A real PR rides the pipeline: feature branch → PR → CI → merge → deploy.yml → dev gateway.
1
One-time: the sample project must be committed to develop (the We-do left it untracked).
2
Branch feature/tweak-view off develop, change the view's height, push.
3
Open a PR into develop. Watch ci.yml run on ubuntu-latest (free): JSON, .deployignore and workflow validation.
4
Merge. deploy.yml fires (note the paths: filter, since docs-only pushes don't deploy). Watch the Ship and Trigger gateway scan steps.
5
Verify at :8089. The dev gateway shows your new height.
# 1 (one-time): get sample onto develop
git add projects/sample
git commit -m "Add sample project"
git push origin develop

# 2: the feature
git checkout develop
git checkout -b feature/tweak-view
# edit …/views/Hello/view.json to change height
git commit -am "Tweak Hello view height"
git push -u origin feature/tweak-view
# then: PR into develop → CI green → merge
Nothing deployed?

Two classic causes: you merged into main instead of develop (only develop triggers deploy.yml), or your fork has no develop branch so the PR silently targeted main.

File-based deploy · you do · part 4
PART 4
Release to prod: main + a tag
10 min. The Git Flow release: bring develop to main, then tag. The tag, not the merge, is what ships.
6
Merge develop into main (a simple --no-ff merge stands in for a release/* branch today).
7
Observe: nothing deploys. A push to main triggers no workflow, by design.
8
Tag v0.1.0 and push the tag. release.yml fires: same five steps, different environment, prod container.
9
Verify at :8090. Prod now shows the change you merged through develop and just released.
git checkout main && git pull
git merge --no-ff develop -m "Release v0.1.0"
git push origin main          # ← nothing happens

git tag v0.1.0
git push origin v0.1.0        # ← release.yml fires
Why tags

Prod always runs a named, re-deployable version. That's also your rollback button: release.yml's workflow_dispatch takes a tag input, so re-deploying v0.1.0 is two clicks, no git surgery.

File-based deploy · you do · part 5
PART 5
Break a deploy on purpose
~5 min, optional but strongly encouraged, because reading the failure is half the lesson. Pick one.
A
Wrong API key. Set garbage in lab-gateway-dev's secret, re-run the deploy. The Ship step succeeds (docker cp doesn't care about keys) but the scan 403s. Files are in the container; the gateway never reloaded. What's your recovery story?
B
Stopped container. docker compose stop ignition-dev, then trigger a deploy. The verify step fails fast: "container is in state 'exited', expected 'running'", before any file moved. Much better than failing halfway.
C
Then fix it and re-run the workflow (manual workflow_dispatch works too). The steps are idempotent, so the deploy converges.
# failure B: stop the target, watch verify fail fast
docker compose stop ignition-dev
# … re-run deploy.yml from the Actions tab …
docker compose up -d ignition-dev  # recovery
Write it down

For your chosen failure: symptom → state of the gateway → recovery → which workflow step should have caught it earlier? Bring the answer to the debrief.

OPTIONAL · 2
Stretch: the modules.json puzzle
Gateway config ships the same way projects do… but not every gateway change rides that path. Two things to discover.
1
It doesn't ship. services/modules.json is a sibling of services/config/, and the workflows docker cp ./services/config/. only. Open deploy.yml and confirm from the docker cp targets. (In this lab it's bind-mounted into all three gateways instead.)
2
A scan can't apply it anyway. Even with a new modules.json on the gateway, module enable/disable needs a restart (docker compose restart ignition-dev), unlike views and DB connections, which hot-reload.
3
Chew on this: why does copy + scan work beautifully for views and database connections, but not for modules? List everything else a scan could never apply (memory, Java args, module binaries…) and bring the list to the debrief.
# deploy.yml ships exactly two trees:
docker cp ./projects/. …:$DATA/projects/
docker cp ./services/config/. …:$DATA/config/
# modules.json is not under either one.
Restart ≠ scan

A restart drops Perspective sessions and takes the gateway down for a minute or more. Deploys that need one are a different beast: plan them, announce them, keep them rare, and keep the daily scan lane free of them.
