You have used Ignition since Lab 02, but always through Git and the UI, never looking at what the gateway writes to disk. Today we open that box. First we decode the 8.3 data/ file tree, then we build a deploy pattern directly on top of it.

Why 8.3 changes the game

In 8.3, gateway configuration lives as JSON resource files on disk (config/resources/…) instead of being locked inside an internal database. Files mean diffs, reviews, and deploys: the whole reason this course can exist in its current shape.

File structure
02
The data/ directory, top to bottom
What you see when you docker exec into a gateway and ls /usr/local/bin/ignition/data. Every entry belongs to one of three buckets.
/usr/local/bin/ignition/data
├── projects/project
│   └── <name>/ (one dir per project)
├── config/resources/gateway
│   ├── <scope>/<module>/<type>/<name>/
│   └── .resources/operational (content-addressed blobs)
├── modules.jsongateway (which modules are enabled)
├── modules/gateway · binaries (installed .modl files)
├── db/operational (config.idb: internal SQLite DB, users included)
└── jar-cache/ · metricsdb/ · var/operational
  • project per-project resources: views, scripts, page config. This is what you deploy daily.
  • gateway cross-project config: DB connections, identity providers, tag providers, module enablement. Shared by everything on the gateway.
  • operational runtime state the gateway owns: the internal DB (user store included), caches, generated blobs. Never versioned, never deployed.
In this lab

On the local gateway, projects/ and config/ are bind-mounted from your repo: the file tree on your laptop is the gateway's disk. The operational paths stay hidden in the container's named volume.

File structure · gateway config
03
Gateway config & deployment modes
Those core / loc / dev / prd scopes are Ignition 8.3's deployment modes feature: one config set holds every environment, and the gateway picks a mode at boot. config/resources/<scope>/<module>/<type>/<name>/{config.json, resource.json}, scope first, then module-namespaced. The modes form an inheritance chain on top of core.
core

The base you version and ship: DB connections, identity providers, tag providers, system props. The portable config every environment inherits from.

→
loc / dev / prd

Per-environment overrides. Selected at boot with -Dignition.config.mode=<scope>, exactly how the lab's three gateways differ.

local

Instance-bound. OPC-UA keystores, the gateway's UUID. Copy this across gateways and you clone one box's identity onto another. Treat as operational.

# the override in action (services/config/resources/)
core/ignition/database-connection/TimescaleDB/config.json
  "connectURL": "jdbc:postgresql://timescaledb:5432/ignition"
loc/ignition/database-connection/TimescaleDB/config.json
  …/ignition_loc   # same name, per-env override
dev/… → …/ignition_dev   ·   prd/… → …/ignition_prd
loc/config-mode.json   # scope descriptor {parent: "core"}

# how a gateway picks its mode: a JVM arg at boot (docker-compose.yaml)
-Dignition.config.mode=loc   # dev/prd gateways boot with dev / prd
The mental model: same name, different reality

In this repo, loc / dev / prd each hold exactly one override: the TimescaleDB connection's connectURL. A view queries "TimescaleDB", and which database that hits depends on the boot mode. The real-plant version: a device named PLC-01 is a simulator in dev and the real PLC in prod, same name, so your project never changes. One config set, one gateway backup, every environment.

File structure · the never list
04
What should never be committed
The operational bucket, spelled out. If any of these ever show up in git status, your ignore rules are broken.
PathWhat it isWhy it's radioactive
config/resources/.resources/Content-addressed blob store (SHA-256 names)Generated and churning, and it lives inside the versioned config tree. The gateway rebuilds it; committing it doubles your repo for nothing.
config/ignition/tags/*.idb · *.digest.json · migration-log-*.mdMore churn inside versioned treesTag value store, theme/font/icon digests, 8.3 migration logs. The shipped .gitignore masks all of these with **/ patterns.
jar-cache/ · metricsdb/ · var/Runtime breadcrumbsRegenerable noise. Often not even worth backing up. (Gateway logs live outside data/, at the install root.)
config/resources/local/Instance identity (OPC-UA keystores, UUID)Belongs to this specific box. Promote it and you clone a gateway's identity onto another.
Where users actually live

There is no separate users file: the internal user store, password hashes included, rides inside db/config.idb, and inside every gwbk backup. Put either in git and you've committed credentials. The versionable pattern for users is an identity provider, which is ordinary config under config/resources/core/ignition/identity-provider/.

File structure · living with the gateway
05
The gateway writes back: taming the churn
On local, your repo is the gateway's disk. That cuts both ways: every UI interaction rewrites manifests inside your working tree, usually touching nothing but a timestamp and a signature. The fix is to undo that noise, not to hide your files from git.
  • Diffs stay clean by construction. .gitattributes routes every resource.json through a normalizer that strips the volatile metadata (timestamp, actor, signature), so git diff only shows changes that matter.
  • Volatile-only rewrites get undone. scripts/clean-ignition-resource-churn.sh compares each modified manifest against HEAD with the metadata stripped; if nothing real changed, --apply restores it. Real edits (and anything staged) are never touched.
  • One exception: machine identity. local-system-properties/config.json belongs to this specific box, so the post-checkout / post-merge / post-rewrite hooks keep just that file skip-worktree. New files the gateway creates stay untracked until you git add them.
# one-time wiring (run once after cloning):
scripts/setup.sh
#   → scripts/git-hooks/       skip-worktree, machine-local file only
#   → scripts/git-diff/normalize-ignition-resource-json.py
#     textconv driver: clean diffs

# undo volatile-only manifest rewrites:
scripts/clean-ignition-resource-churn.sh          # dry run
scripts/clean-ignition-resource-churn.sh --apply  # restore from HEAD
Symptom

"git status shows lots of resource.json changes" → run scripts/clean-ignition-resource-churn.sh, check the list, then --apply. Diffs still full of timestamps, or the machine-local file dirty? Re-run scripts/setup.sh. Full story in docs/TROUBLESHOOTING.md.

File structure → deploy · the hinge
06
Scan or restart? The question that shapes your pipeline
The gateway can hot-reload most file changes when you ask it to re-scan. But not all of them, and the ones it can't are exactly where file-based deploys stop.
ChangeApplied by…
Add / modify a Perspective view✓ project scan
Add / modify a project script✓ project scan
Add / modify a tag UDT✓ scan
Add a database connection✓ config scan
Enable / disable a module (modules.json)✗ gateway restart
Change gateway memory / Java args✗ gateway restart
Remember this line

Scan owns everything that changes daily (views, scripts, connections). When a scan isn't enough, the pipeline needs an extra, disruptive step: docker compose restart ignition-dev in this lab, systemctl restart on a real host. You'll hit the modules.json case yourself in the deploy stretch.

The file-based deploy in five steps
This is literally what deploy.yml does on every push to main, and what release.yml does on every tag.

File-based deploy · pruning
08
.deployignore: what not to put on a gateway
But what if you don't want to deploy everything that is in git? .gitignore won't cut it (those files aren't even in the repo), so we build .deployignore: things that live in git but must never land on a gateway. Think simulation PLCs, tags, and environment-specific config, plus docs and lab tooling.
# .deployignore (shipped in this repo, abridged)
README.md
LICENSE
.github/
docs/
exercises/
instructor-notes/
tests/
third-party-modules/
db-init/
services/config/local/   # ← instance-bound scope!
.DS_Store
docker-compose.yaml # ← unless you want it to run there with your settings
.env
  • Mostly hygiene: a stray README.md inside data/projects/ is harmless noise, but noise accumulates, and some lab files could shadow real resources.
  • Sometimes safety: .env holds your API keys. services/config/local/ is instance identity. Neither may ever reach a shared gateway.
  • The lab's prune step handles literal paths and bare-name globs, not full gitignore semantics (no ! negation). Keep patterns simple.
  • Tailor it per customer. One customer wants simulation tags shipped, another never; one keeps a device config in the repo but off the plant. A per-project .deployignore lets each pipeline draw its own line between "in git" and "on the gateway".
The ignore-file family

.gitignore and .dockerignore are real: Git and Docker read them natively. .deployignore is our invention borrowing their syntax: only the prune step in our workflow knows it exists. Steal the pattern for your own pipelines, but don't expect any tool to honor it for free.

File-based deploy · the ship step
09
Ship = wipe, then copy
dev and prod use named volumes, with no shared filesystem with the runner. So the runner reaches inside the container with docker exec + docker cp.
# the "Ship" step from deploy.yml ($DATA = GATEWAY_DATA_PATH)
docker exec "$IGNITION_CONTAINER" sh -c '
  rm -rf $DATA/projects/*
  # wipe config/, but keep the identity dirs .deployignore protects
  find $DATA/config -mindepth 1 -maxdepth 1 \
    ! -name local ! -name resources -exec rm -rf {} +
  find $DATA/config/resources -mindepth 1 -maxdepth 1 \
    ! -name local -exec rm -rf {} +'

docker cp ./projects/.        "$IGNITION_CONTAINER:$DATA/projects/"
docker cp ./services/config/. "$IGNITION_CONTAINER:$DATA/config/"
  • Why wipe first? docker cp has no --delete. Without the wipe, a view you deleted from the repo keeps living on the gateway, forever.
  • Wipe-then-copy gives you "deleted in repo → deleted on gateway": the same semantics rsync --delete would give a bind-mount setup.
  • Wipe repo-owned, spare gateway-owned. The two dirs kept back, config/local/ and config/resources/local/ (UUID, OPC-UA keystores), are exactly what .deployignore prunes, so they're never copied back. Wiping them would strip the gateway's identity. Everything else, including the scan API token, is committed and copied fresh.
  • It is not atomic. A runner killed mid-copy leaves a partial tree. Two mitigations here. Deploys are idempotent, so you re-run and it converges, and the workflows' concurrency group queues instead of cancelling, so CI never kills its own copy.
In production

The docker cp trick is lab topology. Real-world equivalents: an NFS-mounted data/, or a runner on the gateway host writing to the filesystem directly. Same five steps, different transport.

Don't wipe what production owns

Only wipe paths the repo is the source of truth for. If you manage tags directly in production (edited on the live gateway, not in git), a blanket rm -rf of that tree deletes them, and the copy won't bring them back. Scope the wipe to what you actually ship, or leave gateway-owned resources out of the deploy path entirely.

File-based deploy · the scan API
10
The scan API: one POST to reload the gateway
Files on disk are invisible until the gateway looks. The 8.3 scan endpoints make it look, authenticated with a gateway-issued API key.
# what scripts/scan.sh does under the hood
curl -X POST \
  -H "X-Ignition-API-Token: $IGNITION_API_KEY" \
  http://localhost:8088/data/api/v1/scan/projects

curl -X POST \
  -H "X-Ignition-API-Token: $IGNITION_API_KEY" \
  http://localhost:8088/data/api/v1/scan/config

# the lab wrapper picks key + URL per gateway from .env
scripts/scan.sh both --gateway dev
  • Keys are per-gateway, not per-URL. Generate each in that gateway's UI (Config → Security → API Keys). The dev key will never authenticate against prod.
  • Least privilege: scope the deploy key to Project Scan + Config Scan only. The deploy runner has no business taking backups or writing tags.
  • In CI the key lives as an environment-scoped GitHub secret (lab-gateway-dev / lab-gateway-prod): per-target secrets and deploy history for free, exactly like Lab 03 taught.
Symptom dictionary

"The file is right there but my change isn't picked up" → you forgot the scan. "Scan returns 403" → wrong key, or the key lacks scan permissions. And fittingly: the key you create is itself a config resource on disk at config/resources/core/ignition/api-token/<name>/config.json, storing only a hash.

File-based deploy · where it bites
11
Where people get it wrong
Five classic failure modes, and you'll trigger at least one on purpose in the You-do, because reading the failure is half the lesson.
Copying without wiping

Deleted-in-repo files survive on the gateway and keep being served. Wipe first, always.

Forgetting the scan

Files on disk, gateway oblivious. Runtime serves stale state until someone asks it to look.

Wrong API key for the gateway

The docker cp succeeds (it doesn't care about keys), the scan 403s: files shipped, gateway never reloaded. Know your recovery: fix the secret, re-run, idempotent.

Scanning when a restart is needed

modules.json, memory, Java args: a scan can't apply these. The deploy "succeeds" and nothing changes.

Partial ship (runner dies mid-copy)

The gateway scans a half-state, and a view may reference a script that hasn't landed. Rare, recoverable: re-run to converge. A mature variant stages to a temp dir and swaps; see docs/file-based-deploy-pattern.md.

…and the guard rails in the shipped workflows

Verify step fails fast (container running? key present?) · concurrency group queues instead of cancelling · smoke-check catches a deploy that broke the gateway.
